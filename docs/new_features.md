# New Features — johnboen.com Portal Site

Ideas for evolving the portal based on the dual purpose of the projects (personal growth and value to others), the intent to use the site as both a personal resource and a technology portfolio, and the professional positioning in johnboen.md.

---

**Reframe the hero around the dual purpose.** The current hero reads like a generic project listing. It should communicate that these are living tools the author actually uses — not demo projects or homework assignments. Something closer to: "I build these for myself first, then make them useful to anyone who finds them." This framing is more honest, more interesting, and better explains why a senior engineer with 30 years of experience is publishing certification study tools and story structure explorers. It reframes the projects from "portfolio pieces" into evidence of a working process.

**Strengthen the About section with real professional depth.** The current About is three short sentences that could describe anyone. The johnboen.md content has substance that the portal should reflect: 30 years in database and data engineering, a deliberate pivot into applied AI over the past two-plus years, a working process built around agentic workflows and AI-assisted development, and specific infrastructure like local model hosting. The About section doesn't need to become a resume, but it should give a visitor enough to understand that these projects come from deep experience, not casual interest. A few well-chosen sentences about the engineering philosophy — structured systems, visible pipelines, AI as a daily tool — would make the portfolio angle land much harder.

**Add a "How I Build" section.** The projects demonstrate technologies, but the portal never says so explicitly. A short section explaining the engineering approach — AI-assisted development with Claude as the primary coding platform, structured pipelines from data to presentation, local model infrastructure, agentic workflows — would tie the projects together in a way the Common Themes section currently doesn't. This directly supports the portfolio use case: an employer or collaborator sees not just what was built, but how and with what. Keep it concise — three or four short points with no jargon walls.

**Add technology tags to each project card.** Each project uses different technologies and demonstrates different skills. The Certification site shows content systems and structured question banks. JobClass shows extraction pipelines, warehouse patterns, and analytics publishing. Story Structure Explorer shows agentic generation, API integration, and client-side persistence. AI Usage Trust Paradox shows data visualization and presentation design. Small tags on each card (e.g., "Pipelines," "Agentic," "API Integration," "Data Viz") would let a technical visitor scan the portfolio angle instantly without clicking through to each project.

**Add a "Why this exists" line to each project card.** The current cards describe what each project is. They don't say why it was built. A single line per card — the personal-growth motivation — would make the projects more compelling. For example, Certification exists because the author wanted a structured study system he could actually use. JobClass exists because he wanted to understand how government job data is organized and what the patterns reveal. Story Structure Explorer exists because he wanted to test whether agentic generation could produce constrained creative output. These "why" lines turn feature descriptions into stories, which is better for both portfolio visitors and casual browsers.

**Add a contact or LinkedIn link.** The site currently links only to GitHub. For the portfolio use case — where the primary audience includes potential employers and collaborators — there should be a way to reach out. A LinkedIn link in the nav or footer, or a simple contact line in the About section, closes the loop. Without it, a hiring manager who likes what they see has to go find John Boen separately, which is unnecessary friction.

**Rethink Common Themes as a positioning section.** The current Common Themes section lists abstract principles (Structured Data, Visible Systems, etc.) that feel detached from both the projects and the person. It could be more effective as a section that directly connects the engineering philosophy to the work — something like "What ties these together" that speaks in first person about why every project follows the same patterns: real data in, visible pipeline, usable output. This is the part of the page where a visitor should think "this person has a point of view about how to build things," which is the strongest possible portfolio signal.

**Add a "What's Next" or roadmap indicator.** The user intends to expand these projects and add new ones. A small section or a note at the bottom of the projects area signaling that this is ongoing work — not a finished archive — reinforces the personal-growth angle and tells visitors the site is alive. It doesn't need to be a formal roadmap. Something like "More projects in progress" or a single sentence about the direction of upcoming work is enough.

**Consider card expansion for depth on click.** The main page should stay scannable, but interested visitors — especially technical ones evaluating the portfolio — may want more detail without leaving the portal. An expandable card or a slide-out panel that shows architecture notes, a screenshot, the tech stack, and a "what I learned" paragraph would add depth without cluttering the overview. This is a heavier lift than the other ideas and should come after the content and positioning changes are solid.

**Add a "Built with AI" transparency note.** Given the professional positioning around AI-assisted development, the portal itself is an example of the process. A small footer note or a line in the About section acknowledging that the site and its projects are built using AI tools (Claude, local models, agentic workflows) is both honest and reinforcing. It turns the portal from a page that talks about AI-assisted development into evidence of it.

**Surface freshness — show that projects are maintained.** A "last updated" date or a small activity indicator on each card would signal that these are living projects, not abandoned repos. This matters for both use cases: personal resources need to be current, and a portfolio of stale projects sends the wrong message. This could be as simple as a manually updated date on each card, or as automated as a build step that pulls the latest commit date from each repo.
